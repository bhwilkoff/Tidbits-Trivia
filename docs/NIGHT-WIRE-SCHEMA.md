# Trivia Night — Wire Schema (normative)

The cross-platform contract for networked Trivia Night (Decision 033/v2, see
`docs/CROSS-PLATFORM-MULTIPLAYER.md` for the why). **Two independent
implementations speak this wire** — Swift (`TidbitsTrivia/Core/Networking/
NightProtocol.swift` + `NightTransport.swift`) and Kotlin
(`android/.../net/NightProtocol.kt`) — and they drift silently unless every
change lands in both AND in the golden tests below. Treat this file like
`DATA-CONTRACT.md`: change the wire → change this doc → change both mirrors →
run the golden suite.

## Transport-independence

Everything below is link-agnostic. A transport (Bonjour mDNS+TCP today; Wi-Fi
Aware / BLE / remote later) moves **opaque framed bytes** and never sees
plaintext — Apple behind `NightPeerLink` (`NightLink.swift`), Android behind
`NightPeer` (`NightTransport.kt`).

## Discovery (the default mDNS+TCP link)

- DNS-SD service type: `_tidbits-night._tcp`
- Instance name: `Tidbits-<CODE>`; joiners match by **uppercase suffix** = code
- Room code: 4 chars from `ABCDEFGHJKLMNPQRSTUVWXYZ23456789` (no 0/O/1/I),
  generated by a CSPRNG
- Apple advertises with `includePeerToPeer = true` (AWDL + LAN); Android
  registers/discovers via NSD on the LAN

## Framing + crypto

```
frame = 4-byte big-endian length ‖ body        (length = len(body), ≤ 1 MiB)
body  = nonce(12) ‖ AES-256-GCM ciphertext ‖ tag(16)
key   = SHA256("tidbits-night-v1:" + uppercase(roomCode))
```

- A frame whose GCM tag fails to open is **dropped silently** — this is the
  pairing gate ("only a device that can read the code pairs").
- A length prefix > 1 MiB means a corrupt stream: **drop the whole buffer**
  (resync), not just the frame.
- Nonce is random per message; never reused, never derived.

## Messages

One JSON object per frame. `kind` is required; everything else optional.

| kind | direction | carries |
|---|---|---|
| `join` | joiner → host | `displayName?`, `deviceID?` |
| `answered` | joiner → host | `score` (running TOTAL), `correct` (this question) |
| `leave` | joiner → host | — |
| `welcome` | host → joiner | `seat`, `roomName` |
| `roster` | host → all | `players[]` (full standings, every field emitted) |
| `night` | host → all | `plan`, `questionIds[]`, `questions[]` — see below |
| `begin` | host → all | `questionIndex` |
| `reveal` | host → all | `questionIndex` |
| `finished` | host → all | — |

Types (field names are normative — they are the JSON keys):

- `NightPlayer`: `seat:int`, `name:string`, `score:int`, `answered:bool`,
  `isHost:bool`
- `NightPlan`: `rounds:[{kind:string, count:int}]`, `teams:[string]`.
  `rounds[].kind` is a **GameMode raw value** (`classic`, `pictureId`,
  `thisOrThat`, `closestCall`, `ordering`, `matching`, `typeAnswer`,
  `oddOneOut`, `enumerate`, …) — Swift decodes it as an enum, so an unknown
  kind fails the whole plan. Adding a mode = both engines first, wire second.
- `WireQuestion`: `id`, `prompt`, `options[]`, `correctIndex`, `categoryId`,
  `difficulty`, `explanation`, `sourceTitle`, `sourceUrl`, plus optional
  `imageUrl`, `closest{answer,min,max,step,tolerance,unit}`, `ordering[]`,
  `matching{keys[],values[]}`, `accepted[]`, `enumerate{groups[][]}`,
  `roundIndex`. Note `categoryId`/`sourceUrl` casing — this is the wire DTO,
  NOT Apple's `Question` (`categoryID`/`sourceURL`).

### The `.night` payload — MUST ship both ids and questions

The host sends `questionIds` (the id-based night: each device resolves against
its own corpus — the ~100× smaller payload) **and** full `questions` (the
canonical fallback). Both are required today because the resolvers differ:

- **Android** prefers `questionIds` (resolves via `Corpus.byId`), falls back to
  `questions` for any id it lacks (corpus drift).
- **Apple** currently renders from `questions` only (id resolution is a queued
  optimization) — an ids-only night would show an Apple joiner NOTHING.

An id-based-only night becomes legal ONLY after Apple gains id resolution AND
the corpus id-parity check below is wired into the release ritual.

## Encoding rules (the drift class that bit us on hardware)

- **Kotlin**: `Json { ignoreUnknownKeys = true; encodeDefaults = true;
  explicitNulls = false }`. `encodeDefaults = true` is REQUIRED — Apple's
  `NightPlan`/`NightRound` use strict synthesized Codable and THROW on a
  missing `teams`/`count`.
- **Swift**: lenient custom decoders on `NightMessage` / `NightPlayer` /
  `WireQuestion` (missing key → default), STRICT on `NightPlan` / `NightRound`.
  Null/absent optionals are simply omitted (matches `explicitNulls = false`).
- **Forward-compat delta (known, pinned by test):** an unknown `kind` decodes
  to `.unknown` on Apple (message ignored) but fails the enum decode on
  Android (whole frame dropped by `NightFramer`). Net effect is the same —
  **a new kind must never require acknowledgment** — but never repurpose an
  existing kind.

## Golden tests (run after ANY wire change)

```
tools/night-wire/run_golden.sh
```

1. **Apple harness** (`tools/night-wire/apple_golden.swift`, compiled against
   the REAL repo wire files): decodes the canonical fixtures in
   `tools/night-wire/golden/messages/`, asserts their facts, writes
   `golden/frames/apple-*.hex` (framed AES-GCM, room code `GOLD`).
2. **Android `GoldenWireTest`**: same fixtures + same facts, **decodes the
   apple frames** (Swift-encoded → Kotlin-opened), writes `android-*.hex`.
3. **Apple harness again**: decodes the android frames (Kotlin-encoded →
   Swift-opened).
4. **`check_id_parity.py`**: the bundled corpus id set is identical across
   Apple sqlite / web json / Android json, and every per-mode JSON
   (picture/closest/…) is byte-identical in all three bundles — the
   precondition for the id-based night.

The fixtures are the single canonical source (no per-platform copies). Adding
a wire field = extend a fixture + both `assertFacts` implementations; the two
frame directions then prove the real codecs agree. The `.hex` frames are
regenerated by each run (committed only as a convenience — the fixtures are
the truth).
