## 2. Architecture

**One engine, four mirrors (Decision 019).** The question logic exists in four
places that MUST stay in lockstep — same shapes, same gates, same
FNV-1a/SplitMix64 constants (so the deterministic Daily matches everywhere):

- `tools/corpus/generate_corpus.py` — summary-shape generator (the offline corpus)
- `tools/corpus/wikidata.py` — Wikidata structured-type generator (dataset-driven, cached)
- `TidbitsTrivia/Core/Engine/TemplateEngine.swift` — Apple live path
- `js/engine.js` — Web live path
- `android/.../data/Tidbits.kt` (`TemplateEngine`) — Android live path

**Two pipelines feeding the clients:**
- **Offline corpus** (the bulk + all 22 types): generated by the two python
  scripts → `corpus.sqlite` + `corpus.json`, bundled in each app. Never-repeat,
  offline, fast. Every type is a **4-option MCQ**, so a new type added to the
  corpus renders on ALL platforms with zero client changes (Decision 025).
- **Live generation**: `WikipediaClient` + `TemplateEngine` (Swift/JS/Kotlin
  mirrors) build summary-shape questions from any topic at runtime — powers
  "create a quiz" + corpus-exhaustion fallback. (Live path is summary-shapes
  only; the structured Wikidata types live in the corpus.)

**Apple universal target** (`TidbitsTrivia.xcodeproj`, generated by xcodegen
from `project.yml`): `Core/` (platform-agnostic) + `iOS/` + `tvOS/` behind
`#if os(...)`. **Android** is a separate Kotlin/Compose module in `android/`.
**Web** is vanilla HTML/JS in `/`, `css/`, `js/`.

## 3. Build / run / verify — every platform

Toolchain env (this machine): Xcode is **not** the default; use Xcode-beta via
`DEVELOPER_DIR`. Java for Gradle is Android Studio's bundled JBR.

### iOS
```bash
cd <repo>
xcodegen generate            # ONLY after adding/removing source files
DEVELOPER_DIR=/Applications/Xcode-beta.app/Contents/Developer \
  xcodebuild build -project TidbitsTrivia.xcodeproj -scheme TidbitsTrivia \
  -destination 'platform=iOS Simulator,name=iPhone 17 Pro' -derivedDataPath /tmp/tidbits-dd
# install + run
xcrun simctl boot "iPhone 17 Pro"; xcrun simctl install "iPhone 17 Pro" <app>; \
  xcrun simctl launch "iPhone 17 Pro" com.learningischange.tidbitstrivia
```
### tvOS  (use the device UDID — the name is ambiguous)
```bash
DEVELOPER_DIR=/Applications/Xcode-beta.app/Contents/Developer \
  xcodebuild build -project TidbitsTrivia.xcodeproj -scheme TidbitsTrivia \
  -destination 'platform=tvOS Simulator,id=<Apple TV 4K UDID>' -derivedDataPath /tmp/tidbits-tvdd
```
### Android
```bash
cd android
export JAVA_HOME="/Applications/Android Studio.app/Contents/jbr/Contents/Home"
export ANDROID_HOME="$HOME/Library/Android/sdk"
./gradlew :app:assembleDebug --no-daemon
# emulator: $ANDROID_HOME/emulator/emulator -avd Pixel_9_Pro -no-snapshot-save -gpu swiftshader_indirect
$ANDROID_HOME/platform-tools/adb install -r app/build/outputs/apk/debug/app-debug.apk
$ANDROID_HOME/platform-tools/adb shell am start -n com.learningischange.tidbitstrivia.debug/com.learningischange.tidbitstrivia.MainActivity
```

### Web
```bash

```

**Screenshot / CI hooks** (no-ops in production; env vars):
`TIDBITS_AUTOPLAY=classic:science` (launch into a game), `TIDBITS_AUTOPILOT=1`
(auto-answer to reach reveal/results), `TIDBITS_TAB=records|create`,
`TIDBITS_PARTY=1`, `TIDBITS_ONBOARD=1`, `TIDBITS_AUTOCREATE="<topic>"`. Pass via
`SIMCTL_CHILD_<VAR>` to `simctl launch`. Web equivalents: route via `#/daily`
etc. Screenshots > 2000px tall must be downscaled (`sips -Z 1400`) before Read.

**Discipline:** boot ONE simulator/emulator at a time (never an iOS sim and the
Android emulator together). "Builds" ≠ "works" — always drive the app and look.

## 4. Corpus pipeline (how to change questions)

```bash
cd tools/corpus
python3 -u generate_corpus.py            # regenerate the summary-shape questions
python3 -u wikidata.py                    # regenerate Wikidata types (cache-only, instant)
python3 -u wikidata.py --fetch            # allow LIVE WDQS fetch for un-cached datasets
python3 export_json.py                    # corpus.sqlite → assets/corpus.json (stamps a version)
cp assets/corpus.json android/app/src/main/assets/corpus.json   # sync Android (run from repo root path)
# then rebuild iOS/tvOS/Android; web auto-updates (network-first)
```
Notes:
- `generate_corpus.py` preserves `wd:*` rows; `wikidata.py` replaces only the
  `wd:*` types it produces (targeted DELETE) and **skips un-cached datasets**
  unless `--fetch`. Datasets cached in `tools/corpus/cache/*.json`.
- **WDQS rate-limits HARD** (~1000s `Retry-After`). The cache means you almost
  never re-fetch; only delete a `cache/*.json` to force a refresh of that
  dataset, and run `--fetch` when the endpoint is cool.
- After ANY corpus change, re-run `export_json.py` AND re-sync the Android
  asset, or those two platforms ship stale.
- Gate rules + the no-tells distractor checklist live in
  `docs/QUESTION-QUALITY.md`; change a gate in ALL relevant mirrors at once.

## 5. Toolchain quirks & traps (each cost real time)

- **Xcode**: default dev dir is Command Line Tools → prefix builds with
  `DEVELOPER_DIR=/Applications/Xcode-beta.app/Contents/Developer`. `.xcodeproj`
  is generated by xcodegen from `project.yml` — re-run `xcodegen generate`
  after adding files; never hand-edit the pbxproj.
- **Swift 6 / MainActor-default isolation**: pure Core value types used from
  `nonisolated` engine/DB code are marked `nonisolated` (Question, SeededRNG…).
- **SwiftData store**: plain on-disk on iOS; **Caches-based on tvOS** (App
  Support is not writable on real Apple TV — Decision 017). App-Group
  `groupContainer:` **traps (not throws)** without the entitlement.
- **SourceKit phantom errors** ("Cannot find type X") are stale-index noise —
  trust `xcodebuild`, not the editor squiggles.
- **Android**: Gradle wrapper must be ≥ 9.4.1 (AGP 9.2 floor); `kotlinOptions`
  DSL removed → use `kotlin { compilerOptions { jvmTarget } }`; `compileSdk = 37`
  (Compose 1.12-alpha floor); XML theme parent must be platform
  (`@android:style/Theme.DeviceDefault.Light.NoActionBar`) since there's no
  View-Material lib. v1 is manual DI + sealed `Route` + `BackHandler` (no Hilt/
  Nav3/Room/Ktor); corpus is in-memory from the JSON asset.
- **Web caching** (the bug that hid corpus updates): client is **network-first**
  with IndexedDB as offline-only fallback keyed on the corpus `version`; the
  **service worker is network-first** for corpus/JS/CSS/HTML and cache-first for
  static assets. **Bump `CACHE` in `sw.js` on any shell/code deploy** so the
  worker re-installs.
- **WDQS**: see §4. Heavy per-IP throttle; the dataset cache is the workaround.
