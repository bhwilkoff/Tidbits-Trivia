# /.well-known/ — universal links + app links verification

Place verification files for native-app deep-link binding here.
Served from the domain root (**`https://tidbitstrivia.com/.well-known/…`**) by
GitHub Pages — apex domain via the repo-root `CNAME`, with `.nojekyll` already
present so these files aren't dropped.

> **Status:** the AASA / assetlinks files are NOT committed yet — they require
> the real Apple **TEAMID** + iOS **bundle ID** and the Android **package +
> Play App Signing SHA-256**, which are finalized at first store submission
> (see `store-submission-playbook`). Drop the two files in here at that point;
> the templates below are ready to fill. Tidbits' reserved deep-link path is
> `/item/*` (see `DEEP_LINKS.md`).

## iOS — `apple-app-site-association`

Path: `/.well-known/apple-app-site-association`
**No file extension. No `.json` suffix. Served as
`application/json`.**

```json
{
  "applinks": {
    "apps": [],
    "details": [
      {
        "appID": "TEAMID.com.example.appname",
        "paths": [ "/item/*" ]
      }
    ]
  }
}
```

Replace `TEAMID` with your Apple Developer team ID (App Store
Connect → Membership). Replace `com.example.appname` with your
bundle ID.

## Android — `assetlinks.json`

Path: `/.well-known/assetlinks.json`

```json
[{
  "relation": ["delegate_permission/common.handle_all_urls"],
  "target": {
    "namespace": "android_app",
    "package_name": "com.example.appname",
    "sha256_cert_fingerprints": [
      "UPLOAD_KEY_SHA256_FINGERPRINT",
      "PLAY_APP_SIGNING_KEY_SHA256_FINGERPRINT"
    ]
  }
}]
```

**Include BOTH fingerprints.** The upload key signs your internal-
testing builds; the Play App Signing key (generated by Google after
your first AAB upload — Play Console → Setup → App integrity)
signs the production builds users actually install. Missing either
breaks autoVerify on the corresponding build variant.

**Android 15+ re-fetches `assetlinks.json` every ~24h on devices
with Play Services** — you can rotate fingerprints server-side
without an app update.

## Jekyll users

GitHub Pages runs Jekyll by default and EXCLUDES the `_`-prefixed
and dotfile directories from the published site. Add to `_config.yml`:

```yaml
include:
  - .well-known
```

Without that line, both files 404 silently and `autoVerify` fails
with no error in the app.

## Verifying

After deploy:
- iOS: https://search.apple.com/.well-known/apple-app-site-association
       (Apple's CDN verifies the format)
- Android: https://developers.google.com/digital-asset-links/tools/generator
           (Google's tester)

Without these in place, deep links open the disambiguation chooser
on Android and fall back to the website on iOS — the same broken-
UX symptom for both platforms.
